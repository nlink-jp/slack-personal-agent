# RFP: slack-personal-agent (spa)

> Generated: 2026-04-22
> Status: Draft

## 1. 問題定義

Slack ワークスペースでは、複数のチャネルに情報が分散して流れ続ける。ユーザーは全チャネルを常時追えず、重要な情報の見落としや、過去の議論の検索困難が発生する。既存の Slack ボット（sai）はワークスペース全体の共有エージェントであり、(1) チャネル間の情報分離がない（RAG がグローバルスコープ）、(2) 時間の概念がなく相対時間クエリが破綻する、(3) パーソナルな視点を持てない、という根本的な制約があった。

slack-personal-agent (spa) は、**個人ユーザーの視点**で**複数の Slack ワークスペース**にまたがる複数チャネルを自動監視・情報蓄積し、時間軸を含むチャネルスコープ付き RAG で照会できるパーソナルナレッジエージェントである。3層の情報分離モデル（チャネル → ワークスペース内クロスチャネル → クロスワークスペース）により情報漏えいを防ぎ、ユーザーの明示的許可に基づく知識統合と、承認ゲート付き代理応答を提供する。対象ユーザーは、複数の Slack ワークスペースで多数のチャネルの情報を日常的に追う必要があるチームメンバー。

## 2. 機能仕様

### GUI インターフェース

Wails v2 + React によるデスクトップ GUI アプリケーション。

- チャネル状況ダッシュボード（監視中チャネルの一覧、更新状況、蓄積量の可視化）
- 時間認識付きナレッジ照会インターフェース
- 代理応答の確認・承認・拒否 UI
- チャネル横断許可の管理 UI
- 内部知識ベースの登録・管理 UI

### 入出力

- **入力**: Slack Conversations API からのメッセージデータ（JSON）、添付ファイル、ユーザー登録の内部知識
- **出力**: GUI 上でのダイジェスト表示、RAG 応答、代理投稿（User Token による Slack 投稿）

### 設定

- TOML 設定ファイル（`~/Library/Application Support/slack-personal-agent/config.toml`）
- 環境変数オーバーライド（`SPA_*` プレフィックス）
- 設定項目:
  - ワークスペース定義（複数）: 名前、Slack User Token（`xoxp-`）
  - 監視対象チャネル（ワークスペースごとに自動検出 + 手動除外）
  - ポーリング間隔（デフォルト値 + チャネル個別設定）
  - LLM バックエンド選択（local / vertex_ai）
  - LLM エンドポイント・モデル設定
  - 知識スコープ許可設定（3層分離モデル）:
    - Level 1: チャネル単位（デフォルト、設定不要）
    - Level 2: ワークスペース内クロスチャネル許可グループ
    - Level 3: クロスワークスペース許可フラグ
  - 代理応答タイムアウト
  - ライフサイクル閾値（Hot→Warm、Warm→Cold の時間）

### クレデンシャル管理

Slack User Token および LLM API キーは **macOS Keychain** に保存し、config.toml にはトークンを一切書かない。

- **ストレージ**: `github.com/zalando/go-keyring`（scli で実績あり）
  - macOS Keychain / Linux libsecret / Windows Credential Manager に対応
- **キー構造**: Service=`slack-personal-agent`, Account=`workspace:<name>` / `llm:<backend>`
- **config.toml にはワークスペース名とメタデータのみ**記録（トークンは含まない）
- **テスト用フォールバック**: 環境変数 `SPA_TOKEN_<WORKSPACE>` による注入（開発・テスト時のみ）
- **テスタビリティ**: `Store` インターフェースで Keychain をモック可能（scli パターン踏襲）

```toml
# config.toml — トークンは Keychain に保存、ここには書かない
[[workspace]]
name = "company-a"

[[workspace]]
name = "company-b"
```

### 外部依存

- **Slack API** — Conversations API（User Token `xoxp-`、Keychain から取得）
- **LLM** — ローカル LLM（OpenAI 互換 API）または Vertex AI Gemini
- **DuckDB** — VSS 拡張付きローカルデータベース
- **macOS 通知** — 代理応答の承認リクエスト通知
- **macOS Keychain** — クレデンシャルのセキュアストレージ

### コア機能詳細

#### チャネル監視とAPIキュー制御

ワークスペースごとのキューによる抽象化レイヤーで全 Slack API コールを一元管理する。

- **ワークスペース単位のキュー**: レート制限はワークスペースごとに独立するため、キューもワークスペースごとに管理
- 通常時: 全監視チャネルを一律間隔でポーリング
- 応答発生時: 該当チャネルに優先キューを動的割り当て
- Tier 3 レート制限（~50 req/min per workspace）内で自動調整
- チャネル数とポーリング間隔から消費レートを計算し、制限超過を事前防止

#### 情報蓄積と3層ライフサイクル

shell-agent / sai で実証済みの3層メモリモデルを採用。

| 層 | 状態 | 内容 |
|----|------|------|
| Hot | 直近の生データ | 原文そのまま保持、全タイムスタンプ付き |
| Warm | 要約済み | LLM による要約、時間範囲付き |
| Cold | アーカイブ | 長期保存用、参照のみ |

全レコードに絶対タイムスタンプを付与。現在時刻をコンテキストに注入し、LLM が相対時間（「昨日」「先週」等）を解決する（shell-agent パターン）。

#### 3層知識分離モデルとチャネルスコープ RAG

システム内の基本単位は**チャネル**である。知識のスコープは以下の3層で管理する:

| レベル | スコープ | デフォルト | 有効化方法 |
|--------|---------|-----------|-----------|
| Level 1 | チャネル内のみ | **有効** | 常に有効（基本単位） |
| Level 2 | 同一ワークスペース内クロスチャネル | 無効 | ユーザーがチャネルグループを定義して許可 |
| Level 3 | ワークスペース横断 | 無効 | ユーザーが明示的に許可した知識のみ |

- DuckDB VSS によるベクトル検索に `workspace_id` + `channel_id` フィルタを必須適用
- Level 1（デフォルト）: 質問が発生したチャネル内の知識のみで回答
- Level 2: ユーザーが許可した同一ワークスペース内チャネルグループで横断検索
- Level 3: ユーザーが明示的にクロスワークスペース共有を許可した知識のみ統合
- 許可されていないスコープの情報は絶対に混入させない
- 内部知識ベースの知識も Level 2 または Level 3 のスコープに割り当て可能

#### 代理応答（MITL）

1. チャネル監視中に、システムが応答すべきと判断した場合に発動
2. 下書きを生成し、macOS 通知 + GUI で表示
3. ユーザーが承認 → User Token で投稿（本人として投稿される）
4. タイムアウトまたは拒否 → 破棄
5. 応答発生チャネルのポーリング優先度を動的に引き上げ

#### 内部知識ベース

- Slack 外の知識をユーザーが手動で登録・管理
- テキスト、ドキュメント等を取り込み可能
- RAG の知識ソースとして Slack メッセージと並列で検索対象

#### 添付ファイル分析

- チャネル内で共有されたファイルの内容を取り込み
- `files:read` スコープでダウンロード → 内容抽出 → ナレッジに統合

## 3. 設計判断

| 判断 | 理由 |
|------|------|
| Go + Wails v2 + React | shell-agent で GUI + LLM + DuckDB の組み合わせを実証済み。構造をシンプルに保つ |
| User Token (`xoxp-`) | パーソナルエージェントとしてユーザー自身の権限で動作。チャネル可視性が Slack API 側で自動担保される |
| ポーリング（Events API ではなく） | User Token では Socket Mode が利用不可。ローカルアプリに HTTP webhook も不適。ポーリングが唯一の現実的選択肢 |
| LLM Backend インターフェース | data-agent パターンを踏襲。`Backend` インターフェースでローカル LLM と Vertex AI を config で切替 |
| DuckDB VSS | lite-rag / gem-rag / shell-agent で実績あり。チャネル ID フィルタ付きベクトル検索が自然に実装可能 |
| MITL 代理応答 | User Token での自動投稿はユーザー本人として投稿されるためリスクが高い。shell-agent の承認ゲートパターンを踏襲 |
| ワークスペース単位 API キュー | Slack API レート制限はワークスペースごとに独立。キューもワークスペース単位で管理し、優先度制御で応答品質と API 消費のバランスを取る |
| 3層知識分離モデル | チャネル（基本単位）→ ワークスペース内クロスチャネル → クロスワークスペースの3段階。デフォルトはチャネル閉じ、明示許可で段階的に開放 |
| 複数ワークスペース対応 | ユーザーは複数の Slack ワークスペースに参加している。ワークスペースごとに User Token と API キューを独立管理 |
| Keychain ファースト | GUI アプリでありヘッドレス動作は不可。トークンは macOS Keychain に保存し config.toml には一切書かない。scli の `go-keyring` パターンを踏襲 |
| 3層ライフサイクル | sai → shell-agent で段階的に改良された記憶モデル。時間認識は shell-agent の実装を基盤とする |

### 既存ツールとの関係

| ツール | 関係 |
|--------|------|
| sai (lab-series) | 記憶モデルと RAG の原型。チャネル分離なし・時間認識なしの課題を解決 |
| shell-agent (util-series) | 時間認識記憶、GUI スタック、MITL、LLM 統合の基盤 |
| data-agent (util-series) | LLM Backend 抽象化パターンの出典 |
| stail (chatops-series) | Socket Mode リアルタイム監視の知見（本プロジェクトではポーリングを採用） |
| gem-rag / lite-rag | DuckDB VSS による RAG の実績 |

### スコープ外

- DM の監視
- マルチユーザー対応（1ユーザーのパーソナルツール）
- Slack 以外の外部情報ソースとの自動連携（内部知識ベースは手動登録）

## 6. シリーズ配置

**Series:** chatops-series

**理由:** Slack ワークスペースの情報収集・自動化が主軸であり、chatops-series（Slack ChatOps 自動化ツール群）のドメインに合致する。swrite / scat / stail / slack-router と同じ Slack エコシステム内のツールとして位置づけられる。

## 7. 外部プラットフォーム制約

### Slack API

| 制約 | 影響 | 対策 |
|------|------|------|
| Tier 3 レート制限（~50 req/min per workspace） | 多チャネル × 複数ワークスペースで制限到達の可能性 | ワークスペース単位キューで消費レート管理 |
| `conversations.history` 上限 1000件/回 | 大量メッセージの初回取得 | cursor ページネーション対応 |
| User Token OAuth フロー | 初回セットアップにブラウザ認証が必要 | セットアップウィザード提供 |
| Token 有効期限 | User Token は長期間有効だが失効の可能性 | エラーハンドリングと再認証フロー |
| ファイルダウンロード | URL にトークンパラメータが必要 | `files:read` スコープ + Authorization ヘッダー |

### macOS

| 制約 | 影響 | 対策 |
|------|------|------|
| 通知権限 | MITL 通知に macOS 通知許可が必要 | 初回起動時に権限リクエスト |
| バックグラウンド実行 | ポーリングの継続性 | macOS サービスとして設計 |

## 議論ログ

### sai の課題分析

既存の sai（lab-series）を評価し、以下の根本的課題を特定:

1. **チャネル分離なし**: RAG 検索がワークスペース全体をスコープとしており、`retriever.retrieve()` にチャネルフィルタがない。#private-engineering の質問に #general の内容が混入しうる
2. **時間認識なし**: メッセージにタイムスタンプはあるが、LLM コンテキストに現在時刻が注入されず、相対時間クエリ（「昨日の議論」等）が破綻
3. **グローバルボット**: Bot Token でワークスペース全体に対応する共有エージェントであり、個人ごとの視点・権限・知識範囲を持てない

→ sai の進化ではなく**完全新規プロジェクト**として設計することを決定。

### shell-agent からの技術継承

shell-agent が sai の記憶概念を取り込み、さらに時間認識を実装していた:

- 現在時刻をシステムプロンプトに注入、Hot メッセージに `[HH:MM:SS]` プレフィックス
- 3層ライフサイクル（Hot/Warm/Cold）+ Pinned メモリ
- LLM が現在時刻とメッセージタイムスタンプの差分から相対時間を解決

→ この時間認識パターンを本プロジェクトの基盤とする。

### 認証モデルの決定

- Bot Token は共有エージェント向きであり、パーソナルエージェントには User Token が適切
- User Token であればユーザーのチャネル可視性が Slack API 側で担保される
- Socket Mode は User Token では利用不可 → ポーリングを採用

### 代理応答の安全設計

- User Token での投稿はユーザー本人として表示されるため、自動投稿のリスクが高い
- shell-agent の MITL パターンを踏襲: システムが応答意図を検知 → macOS 通知 + GUI → ユーザー承認/タイムアウト

### LLM 抽象化の方針

- data-agent の `Backend` インターフェースパターンを採用
- config.toml の `[llm] backend = "local" | "vertex_ai"` で切替
- Slack メッセージをクラウドに送るかどうかはユーザーの判断に委ねる

### スコープの調整

当初スコープ外としていた以下を再検討:

- **内部知識ベース**: Slack 外の知識を手動登録できる機能として追加
- **添付ファイル分析**: 可能であれば実現が望ましいとして Phase 4 に配置

### 複数ワークスペース対応の追加

当初は単一ワークスペースを想定していたが、ユーザーが複数ワークスペースに参加している現実を反映:

- 各ワークスペースに個別の User Token が必要
- API キューはワークスペース単位で独立管理（レート制限がワークスペースごとに独立するため）
- GUI はワークスペースをまたいだ統合ビューを提供

### 3層知識分離モデルの確定

チャネルをシステムの基本単位とし、グローバル知識を2レイヤーで構成:

- **Level 1（デフォルト）**: チャネル内知識のみ — 最も安全、設定不要
- **Level 2**: ワークスペース内クロスチャネル — ユーザーが許可したチャネルグループ間で共有
- **Level 3**: クロスワークスペース — ユーザーが明示的に許可した知識のみワークスペースを越えて利用可能

この設計により、情報漏えいのリスクを最小化しつつ、必要に応じて知識統合の範囲を段階的に拡大できる。
